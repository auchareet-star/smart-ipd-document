📖 0. วิธีอ่านสเปกนี้

แต่ละรายการคือ "ความสามารถเชิงตรรกะ" ที่ HIS ต้องจัดให้ — ไม่บังคับรูปแบบเทคนิค ชื่อ operation เช่น His.GetDrugCatalog เป็นเพียงตัวอ้างอิง HIS เลือกได้ว่าจะส่งมอบผ่านช่องทางไหน

READ Smart IPD อ่านจาก HIS WRITE Smart IPD ส่งกลับเข้า HIS EVENT HIS ส่งเข้ามาเมื่อมีเหตุการณ์ MUST จำเป็น OPT เสริม
  • ช่องทาง 1 เปิดฐานข้อมูล (DB) ให้อ่าน read-only · ช่องทาง 2 เปิด API ของ HIS · ช่องทาง 3 HIS push เข้ามา
  • READ ใช้ช่องทาง 1/2 · WRITE ต้องผ่านช่องทาง 2 (ห้ามเขียน DB ตรง) · EVENT ใช้ช่องทาง 3
  • Master data ปริมาณมากแต่เปลี่ยนช้า → แนะนำ sync เข้ามา cache แทนการเรียกซ้ำ
ขอบเขตความครบถ้วน (v1.1)

สเปกนี้ครอบคลุม ทั้งการอ่าน (READ) และการเขียน (INSERT/UPDATE/DELETE) ที่ "ข้ามไป HIS" จริง — ตรวจทานจาก 362 endpoint ของ Smart IPD (รวม 139 operation ที่เป็น write) เทียบกับ adapter จริง. operation ที่เขียน "ข้อมูลคลินิก IPD" (บันทึก/คำสั่ง/แผน) เป็นการเขียน local ของ Smart IPD ไม่นับเป็นสิ่งที่ HIS ต้องจัด (ดูข้อ 12). ส่วนที่ใช้ POST แต่เป็น "การค้นหา" ถือเป็น READ

🔑 1. รหัสอ้างอิงที่ต้องแนบมาทุกครั้ง

ข้อมูลทุก operation ต้องมีรหัสอ้างอิงเหล่านี้ติดมาด้วย (เป็น "ค่าที่ HIS ใช้จริง" Smart IPD เก็บไว้อ้างอิงโดยไม่ดัดแปลง)

ฟิลด์ความหมาย

drugCode / labCode / xrayCode / icdCodeรหัสรายการในบัญชีกลาง
wardCode / bedCodeรหัสวอร์ด / เตียง

🟢 12. ส่วนที่ Smart IPD เป็นเจ้าของเอง (HIS ไม่ต้องจัด)

เพื่อความชัดเจน — รายการต่อไปนี้ Smart IPD เก็บ/กำหนดเอง HIS ไม่ต้องเตรียม แม้ในระบบเดิมจะเคยมาจาก HIS

รายการทำไมเป็นของ Smart IPD
Smart IPD เอกสารคลินิก IPD ทั้งหมด — บันทึกแพทย์/พยาบาล, แผนการพยาบาล, คำสั่ง (ต้นฉบับ), eMAR, สรุปจำหน่าย, น้ำเข้า-ออก, med reconcile, รับผู้ป่วยใหม่เนื้อหาเกิด/ใช้เฉพาะใน IPD — เก็บในฐานข้อมูล Smart IPD
Smart IPD Master ของน้ำเข้า-ออก (Intake/Output items)เป็น config ของ Smart IPD (เดิม TBM_IPD_INTAKE/OUTPUT)
Smart IPD คลังแม่แบบงานพยาบาล FAIE (focus / assessment / intervention / outcome / nurse-note templates)config เฉพาะงานพยาบาล IPD
Smart IPD code list ของแบบฟอร์มรับผู้ป่วย (วิธีมา รพ., chief complaint type, ความสัมพันธ์, ที่อยู่อาศัย ฯลฯ)enum ของฟอร์ม Smart IPD (ตกลงรายต่อ HIS ได้ — บางรพ.อาจอยากใช้ของ HIS)
Smart IPD Auth/JWT, ผู้ช่วย AI, ป้ายกำกับผู้ป่วย, ไฟล์ภาพ DICOM ในเครื่องฟีเจอร์เฉพาะ Smart IPD

สรุป: HIS จัดเฉพาะ "ข้อมูลกลางของโรงพยาบาล" (ข้อ 2–9) ส่วน "เนื้อหาการดูแลผู้ป่วยใน" Smart IPD ดูแลเอง

🔒 13. Non-functional & Conformance

Non-functional

  • Auth: DB → read-only; API → OAuth2/API key/mTLS; EVENT → verify signature
  • Encryption: HTTPS/TLS · Idempotency: SubmitOrder กันส่งซ้ำด้วย orderId
  • Environments: มี Test/UAT + ข้อมูลตัวอย่าง ก่อน Production
  • Error: ตอบรหัสที่สื่อความหมาย (ไม่พบผู้ป่วย / คำสั่งถูกปฏิเสธ + เหตุผล)
  • Availability: ตกลงเวลาให้บริการ + พฤติกรรมเมื่อ HIS ออฟไลน์ (อ่านจาก cache, WRITE เข้าคิว retry)

Conformance Checklist — ขั้นต่ำเพื่อ Go-Live

  • GetAdmittedPatients + PatientEvents — รายชื่อในวอร์ด + เหตุการณ์ ADT
  • GetPatientInfo / GetPatientAdmissionCard — demographics + แพ้ยา + การรับเข้า
  • นัดหมาย — อ่านปฏิทิน/ช่องเวลา + สร้าง/แก้/ยกเลิก/เลื่อน (ถ้าใช้ฟีเจอร์นัด)
  • Master: วอร์ด/เตียง, ห้อง, urgent type, general order category
  • Master: ICD + diagnosis types + discharge status/type + organs
  • Drug catalog — ค้นยา + timing/route/form/unit + ราคา
  • Lab + Xray catalog — รายการ + group/modality
  • Directory — แพทย์ / พยาบาล
  • GetLabResults / GetXrayResults — ผล + ลิงก์ PACS
  • สัญญาณชีพ — อ่าน (กราฟ/ตาราง) + เขียน (บันทึก/แก้/ลบ) — ถ้า HIS เป็นเจ้าของ
  • SubmitOrder — รับคำสั่ง + ตอบ orderId
  • UpdateOrder + OrderStatusFeedback — แก้/ยกเลิก + แจ้งสถานะกลับ
  • รหัสอ้างอิง + Auth + Test environment
