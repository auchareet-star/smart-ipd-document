1. วัตถุประสงค์และขอบเขต

เป้าหมาย: ทำให้ Smart IPD เป็นผลิตภัณฑ์ที่ "ไม่ผูกกับ HIS เจ้าใดเจ้าหนึ่ง" — โรงพยาบาลที่ใช้ HIS ยี่ห้ออื่น (ไม่ใช่ HAOS/Panacea) สามารถนำ Smart IPD ไปติดตั้งและเชื่อมกับ HIS เดิมของตนได้ โดยไม่ต้องแก้โค้ดแกนของ Smart IPD เพียงเขียน "adapter" หนึ่งตัวต่อ HIS หนึ่งเจ้า

หลักการสำคัญ 3 ข้อ

  1. Smart IPD เป็นเจ้าของ domain ของ IPD เอง — ข้อมูลที่เคยอยู่ในตาราง TB_IPD_*/TBM_IPD_* ของ HAOS จะย้ายมาอยู่ในฐานข้อมูลของ Smart IPD (ดูเอกสาร Ownership Analysis)
  2. HIS เป็น "system of record" ของข้อมูลแกนกลาง — ผู้ป่วย, การ admit, แพทย์/พยาบาล, drug/lab/xray catalog, การเงิน ยังเป็นของ HIS
  3. การคุยกันเกิดผ่าน contract กลางเพียงจุดเดียว — interface IHisProvider (เดิมคือ IHisAdapter) แต่ละ HIS implement contract นี้ตามเทคโนโลยีที่ตัวเองมี
ข้อได้เปรียบ: contract นี้มีอยู่แล้ว

ระบบปัจจุบันมี IHisAdapter (~322 methods แบ่งเป็น 31 domain modules เช่น Patient, OrderNote, Lab, Dispensing) ทำหน้าที่เป็น "seam" คั่นระหว่าง Smart IPD กับ HAOS อยู่แล้ว — DTO ที่ส่งผ่านก็เป็นชนิดกลางของ Smart IPD เอง (เช่น AdmittedPatientDto, HisDoctorNoteDto) ไม่ใช่ schema ของ HAOS. Spec นี้คือการ "ทำให้ contract นี้เป็นมาตรฐานและ vendor-neutral อย่างเป็นทางการ" ไม่ได้สร้างใหม่จากศูนย์

2. หลักการสถาปัตยกรรม: Port–Adapter (Hexagonal)

┌─────────────────────── Smart IPD Product (vendor-neutral core) ───────────────────────┐ │ │ │ Frontend (Angular) → WebAPI Controllers → Application Services │ │ │ │ │ ┌───────────────┴───────────────┐ │ │ │ Smart IPD OWN DB │ │ │ │ (IPD clinical data, auth, │ │ │ │ AI, tags, DICOM index) │ │ │ └───────────────────────────────┘ │ │ │ │ │ ╔═════════ PORT ═══════╧═══════════╗ │ │ ║ IHisProvider (contract กลาง) ║ │ │ ║ ~322 capability methods ║ │ │ ╚═════════╤═══════════╤═══════════╤═╝ │ └──────────────────────────────────────────────┼───────────┼───────────┼──────────────────┘ │ │ │ ┌──────────┴──┐ ┌──────┴─────┐ ┌───┴────────┐ │ HAOS Adapter│ │ HIS-B Adapt│ │ HIS-C Adapt│ ← ADAPTERS │(มีแล้ววันนี้)│ │ (FHIR) │ │ (DB ตรง) │ (1 ตัว/HIS) └──────┬──────┘ └─────┬──────┘ └─────┬──────┘ │ │ │ HAOS HIS HIS เจ้า B HIS เจ้า C
  • Port = IHisProvider — สัญญา (capabilities) ที่ Smart IPD ต้องการจาก "HIS ใดก็ได้" นิยามด้วย DTO กลางของ Smart IPD
  • Adapter = หนึ่ง assembly ต่อหนึ่ง HIS — แปลง (map) ระหว่าง contract กลาง ↔ เทคโนโลยีจริงของ HIS นั้น (REST/SOAP/FHIR/SQL)
  • Smart IPD core ไม่รู้จักว่าปลายทางเป็น HIS เจ้าไหน — เลือก adapter ผ่าน DI + config ตอน runtime
// Smart IPD core เรียกผ่าน contract เดียว — ไม่ผูกยี่ห้อ
public interface IHisProvider {
    Task<AdmittedPatientDto[]> GetAdmittedPatientsAsync(WardFilter f);
    Task<PatientAdmissionCardDto?> GetPatientAdmissionCardAsync(string an);
    Task<string> SubmitMedicationOrderAsync(MedicationOrderDto order); // write-back to HIS
    // ... ~322 capabilities จัดกลุ่มเป็น 31 domain
}

// แต่ละ HIS = adapter หนึ่งตัว เลือกด้วย config
services.AddHisProvider(config["His:Vendor"]); // "haos" | "fhir" | "vendorB"

3. โมเดลความเป็นเจ้าของข้อมูล (Data Ownership — 3 Zones)

การจะตอบว่า "ส่วนไหนมีใน Smart IPD เอง ส่วนไหนต้องใช้จาก HIS" ต้องแบ่งข้อมูลเป็น 3 โซน:

Zone A Smart IPD เป็นเจ้าของ 100% (อยู่ใน DB ของ Smart IPD)

ข้อมูลที่เกิดและใช้เฉพาะใน IPD workflow — ไม่ต้องพึ่ง HIS เลย

  • เอกสารคลินิก IPD: Doctor/Nurse Notes, Order Notes, eMAR (Order Plan), Nursing Care Plan, Admission Record, History & Physical, Discharge Summary, Intake/Output, Med Reconcile (เดิม TB_IPD_*)
  • Master/config ของ IPD: FAIE (focus/assessment/intervention/outcome/templates), I/O master (เดิม TBM_IPD_*)
  • ฟีเจอร์เฉพาะ Smart IPD: Authentication/JWT, AI Chat + cache, Patient Tags, DICOM file index

Zone B ข้อมูลอ้างอิงจาก HIS (อ่านเป็นหลัก — Read)

Smart IPD ต้องดึงมาแสดง/ใช้ประกอบ แต่ไม่ได้เป็นเจ้าของ — HIS คือ system of record

  • Patient demographics, รายการ admit ปัจจุบัน (admitted patients), ward/bed
  • Master data กลาง: ward, ICD, drug catalog, lab/xray catalog, routes, units, ฯลฯ
  • ผลตรวจ: Lab results, X-ray/PACS results, Vital signs (ถ้า HIS เป็นคนเก็บ)
  • Directory บุคลากร: แพทย์, พยาบาล, เภสัชกร
  • ประวัติ OPD/IPD เดิม

Zone C ข้อมูลที่ต้อง "เขียนกลับ" เข้า HIS (Write-back)

เกิดใน Smart IPD แต่ต้องไหลเข้า HISเพื่อระบบกลาง (billing, pharmacy, เวชระเบียนทางกฎหมาย)

  • คำสั่งยา/แล็บ/เอกซเรย์ (orders) → เข้าระบบ pharmacy/LIS/RIS และการเงินของ HIS
  • การจ่ายยา (dispensing) → ตัด stock / charge เข้า TB_FINANCES ของ HIS
  • Diagnosis (ICD) / Discharge status → เวชระเบียน + เคลม
  • (ตัวเลือก) สำเนาเอกสารคลินิกบางส่วน เพื่อรวมศูนย์เวชระเบียน
หัวใจของ Spec

Zone A = "มีใน Smart IPD เอง" · Zone B = "อ่านจาก HIS" · Zone C = "เขียนกลับ HIS". Zone B และ C คือสิ่งที่ contract IHisProvider ต้องครอบคลุม และแต่ละรายการเลือกช่องทางเชื่อมต่อได้ (ข้อ 4)

4. ช่องทางการเชื่อมต่อ (3 Integration Modes)

แต่ละ capability ใน contract สามารถ realize ด้วยช่องทางใดช่องทางหนึ่ง (หรือผสม) — กำหนดได้ต่อ HIS ต่อ capability ผ่าน config

Mode 1 · Direct DB เชื่อม DB ของ HIS ตรง Mode 2 · HIS-Provided API เรียก API ที่ HIS เปิดให้ Mode 3 · Smart IPD-Provided API HIS ส่งข้อมูลเข้า Smart IPD
Modeทิศทางคำอธิบายเหมาะกับข้อควรระวัง
Mode 1
Direct DB		 Smart IPD → DB ของ HIS Adapter query ตาราง/view/stored-proc ของ HIS โดยตรง (อ่านอย่างเดียวแนะนำให้ใช้ read replica). เป็นวิธีที่ HAOS adapter ใช้อยู่จริงวันนี้ ผ่าน HaosDatabaseService + HaosConnection HIS ที่ไม่มี API ครบ / ต้องการ performance สูง / query ซับซ้อนข้ามหลายตาราง / enrichment ผูกกับ schema ภายในของ HIS (เปราะเมื่อ HIS upgrade); เสี่ยงด้านสิทธิ์/ความปลอดภัย; เขียนกลับตรงอันตราย — ควรอ่านเท่านั้น
Mode 2
HIS API		 Smart IPD → API ของ HIS Adapter เรียก REST/SOAP/FHIR/HL7 ที่ HIS provide. HAOS adapter ใช้วิธีนี้ผ่าน Kiota client สำหรับการเขียน (orders, notes) การเขียนกลับ (Zone C) ที่ต้องผ่าน business logic ของ HIS; ข้อมูลที่ HIS มี API พร้อม; ลด coupling กับ schema ขึ้นกับว่า HIS มี API ครบหรือไม่; latency; ต้อง map auth ของแต่ละ HIS
Mode 3
Smart IPD API		 HIS → API ของ Smart IPD Smart IPD เปิด inbound endpoint/webhook ให้ HIS "push" ข้อมูลเข้ามา (เช่น ADT feed แจ้ง admit/discharge/transfer, master data sync). HIS เป็นฝ่ายเรียก ตาม contract ที่ Smart IPD กำหนด เหตุการณ์ real-time (admit/transfer/discharge); การ sync master data เป็นรอบ; HIS ที่ส่ง HL7 ADT/ORM ได้อยู่แล้ว; ลดภาระ polling ต้องมี idempotency + ordering; Smart IPD ต้องเก็บสำเนา (cache/replica) ของข้อมูลที่รับเข้ามา

กฎการเลือก Mode (Decision Guide)

  • เขียนกลับเข้า HIS (Zone C) → ใช้ Mode 2 เป็นหลัก (ผ่าน business logic ของ HIS) — ห้ามเขียน DB ตรง
  • อ่านข้อมูล master/reference ที่เปลี่ยนช้า (Zone B) → Mode 3 sync เข้ามา cache (ดีสุด) หรือ Mode 2 เรียกตามต้องการ
  • อ่านข้อมูลปริมาณมาก/ซับซ้อน หรือ HIS ไม่มี API → Mode 1 (read replica)
  • เหตุการณ์ผู้ป่วย (admit/transfer/discharge) → Mode 3 webhook/ADT ถ้าได้, fallback เป็น polling ผ่าน Mode 1/2

7. Identity & Correlation (การอ้างอิงข้ามระบบ)

เมื่อ Smart IPD เก็บข้อมูล IPD เอง (Zone A) แต่ต้องอ้างถึงผู้ป่วย/ผู้ใช้ของ HIS — ใช้หลัก "External Reference ID แบบ opaque" ไม่ทำ DB-level FK ข้ามฐาน

  • ตาราง Zone A เก็บ HisPatientRef (เช่น HN), HisAdmissionRef (เช่น AN/AdmitId), HisUserRef เป็น "ค่า" (string/identifier ของ HIS)
  • ความหมายของ id เหล่านี้ถูก "ตีความ" โดย adapter เท่านั้น — HIS เจ้าอื่นอาจใช้ AN คนละรูปแบบ; adapter รับผิดชอบ resolve
  • มี correlation map กลาง: ระบบ identifier ของ Smart IPD ↔ identifier ของ HIS (กรณีต้อง normalize)
  • validation เกิดที่ระดับ application (เรียก IHisProvider.PatientExists(an)) ไม่ใช่ที่ DB constraint
Reference ในข้อมูล Smart IPDความหมายใครตีความ
HisAdmissionRef (AN)ตอน admission หนึ่งครั้งของผู้ป่วยadapter ของ HIS นั้น
HisPatientRef (HN)ตัวผู้ป่วยadapter
HisUserRefแพทย์/พยาบาล/เภสัช ผู้ทำรายการadapter + directory sync
HisDrugRef, HisOrderRefยา / order ที่เขียนกลับไปadapter (คืน id หลัง submit)

8. Connection Profile (Configuration-Driven)

การเลือก HIS และช่องทางต่อ capability ทำผ่าน config ทั้งหมด — ไม่ hardcode. ตัวอย่างโครงสร้าง:

// appsettings.{hospital}.json
{
  "His": {
    "Vendor": "haos",            // เลือก adapter assembly
    "Connection": {
      "DbReadOnly": "Server=...;Database=HIS;...",   // Mode 1 (อ่าน)
      "ApiBaseUrl": "https://his-api.hospital.org",    // Mode 2
      "ApiAuth": { "Type": "OAuth2", "ClientId": "...", "Secret": "..." }
    },
    "Inbound": {                    // Mode 3 (HIS push เข้า Smart IPD)
      "AdtWebhookEnabled": true,
      "SharedSecret": "..."
    },
    "CapabilityModes": {           // override ช่องทางต่อ capability
      "GetAdmittedPatients": "InboundPush",
      "GetLabResults": "DirectDb",
      "SubmitMedicationOrder": "HisApi",
      "DrugCatalog": "SyncCache"
    },
    "Features": {                  // ปิด capability ที่ HIS นี้ไม่มี
      "VitalSignsOwnedBySmartIpd": true,
      "DispensingEnabled": false
    }
  }
}
  • Capability ที่ HIS ไม่รองรับ → ปิดด้วย feature flag; Smart IPD ซ่อน/disable UI ส่วนนั้น หรือใช้ fallback (เช่น ถ้าไม่มี vital signs API → ให้ Smart IPD เก็บเอง = ย้ายเข้า Zone A)
  • หนึ่ง deployment = หนึ่งโรงพยาบาล = หนึ่ง profile

9. ขั้นตอนการเชื่อม HIS เจ้าใหม่ (Onboarding Playbook)

  1. ประเมิน capability ของ HIS — เทียบกับ Capability Catalog (ข้อ 5) ว่า HIS มี API/DB/ความสามารถ push อะไรบ้าง → กรอกตาราง gap
  2. เลือก Mode ต่อ capability — ตามกฎข้อ 4 (เขียน=HIS API, อ่านเยอะ=DB, event=push)
  3. สร้าง adapter project ใหม่ เช่น HAOS.SmartIPD.Adapters.VendorB implement IHisProvider (เลียนแบบโครงสร้าง HaosHisAdapter.*.cs ทีละ domain)
  4. เขียน mapping layer — schema/payload ของ HIS ↔ Canonical DTO (ข้อ 6)
  5. ตั้ง identity resolution — กำหนดว่า AN/HN/UserId ของ HIS นี้หน้าตาอย่างไร (ข้อ 7)
  6. กำหนด connection profile (ข้อ 8)
  7. รัน Conformance Test Suite — ชุดทดสอบกลางที่ยิงทุก capability ต้องผ่าน "MUST" ทั้งหมด (ข้อ 11)
  8. UAT + go-live — เปิดทีละ domain (อ่านก่อน เขียนทีหลัง)

หมายเหตุการ migrate ข้อมูล IPD เดิม: สำหรับโรงพยาบาลที่เคยใช้ HAOS — ต้องย้ายข้อมูลใน TB_IPD_*/TBM_IPD_* เข้า DB ของ Smart IPD (เป็น one-time migration). โรงพยาบาลใหม่ที่ใช้ HIS เจ้าอื่น เริ่มจาก DB เปล่าของ Smart IPD ได้เลย

10. Cross-cutting Concerns

10.1 Security & Auth

  • Mode 1 (DB): ใช้ account อ่านอย่างเดียว + read replica; เก็บ credential ใน secret store ไม่ใช่ในโค้ด
  • Mode 2 (HIS API): adapter รับผิดชอบ auth ของแต่ละ HIS (OAuth2 / API key / mTLS / SOAP WS-Security) — abstract ไว้ใน IHisAuthProvider
  • Mode 3 (inbound): verify signature/shared-secret/mTLS ของ payload ที่ HIS ส่งเข้ามา; rate limit
  • Smart IPD ยังคงมี auth ของตัวเอง (JWT) สำหรับผู้ใช้งาน — แยกจาก auth ที่คุยกับ HIS

10.2 Consistency & Sync

  • ข้อมูล Zone B ที่ cache ไว้ (master data, admitted list) ต้องมีกลยุทธ์ invalidate — TTL หรือ event-driven (Mode 3)
  • การเขียน Zone C ต้อง idempotent + เก็บ HisOrderRef ที่ HIS คืนมา เพื่อกัน duplicate
  • กรณี HIS ล่ม: อ่าน (Zone B) ใช้ cache ค้างได้; เขียน (Zone C) เข้า outbox queue แล้ว retry

10.3 Error Handling & Observability

  • นิยาม error กลางใน contract (เช่น HisUnavailable, PatientNotFound, OrderRejected) — adapter แปลง error ของ HIS เป็นชนิดกลาง
  • Log/trace ต่อ capability + ต่อ HIS เพื่อ debug การ integration
  • Circuit breaker / timeout ต่อ Mode 2/3

11. Conformance — ขั้นต่ำที่ HIS ต้องรองรับ

MUST — Minimum Viable Integration (Smart IPD ทำงานพื้นฐานได้)

  • รู้รายชื่อผู้ป่วยที่ admit อยู่ในวอร์ด + demographics + ADT events (admit/transfer/discharge)
  • Master data: ward/bed, drug catalog, lab/xray catalog, ICD, directory แพทย์/พยาบาล
  • อ่าน Lab + Xray results
  • เขียนกลับ orders (med/lab/xray) + รับ status feedback

OPTIONAL — Enhanced

  • Dispensing/charge write-back, expense summary
  • Vital signs (ถ้าไม่มี → Smart IPD เก็บเอง)
  • Diagnosis/discharge write-back, patient picture, report data, OPD/IPD history

Conformance Test Suite: ชุดทดสอบกลาง (1 ชุด ใช้กับทุก adapter) ยิงทุก MUST capability ด้วยข้อมูลจำลอง แล้วตรวจว่าคืน Canonical DTO ถูก shape — adapter ผ่านชุดนี้ = พร้อม go-live

12. สรุป — อะไรอยู่ที่ Smart IPD เอง vs ต้องใช้จาก HIS

ส่วนงานเจ้าของถ้าต้องใช้จาก HIS — ช่องทาง

บรรทัดเดียวจบ

Smart IPD เป็นเจ้าของ "สมุดบันทึกทางคลินิก IPD" ทั้งเล่ม และคุยกับ HIS เจ้าใดก็ได้ผ่าน contract IHisProvider เพื่อ (ก) อ่านบริบทผู้ป่วย/master/ผลตรวจ และ (ข) เขียนกลับ order/charge — โดยแต่ละ HIS เลือกได้ว่าจะให้ Smart IPD ต่อผ่าน DB ตรง, API ของ HIS, หรือ push เข้า API ของ Smart IPD ตามความพร้อมของตน
