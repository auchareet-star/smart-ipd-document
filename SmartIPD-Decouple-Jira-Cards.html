🟪 EPIC — แยก Smart IPD ออกจาก HIS ให้เป็นระบบอิสระ (HIS-agnostic)

Type: Epic · Priority: Highest · Component: SA/BA, Smart IPD, HIS Integration · Labels: decouple his-integration architecture
Goal

ออกแบบให้ Smart IPD เป็นเจ้าของข้อมูลคลินิกผู้ป่วยใน (IPD) เอง และเชื่อมกับ HIS เจ้าใดก็ได้ผ่าน contract มาตรฐานจุดเดียว เพื่อขาย/ติดตั้งกับโรงพยาบาลที่ใช้ HIS เจ้าอื่นได้

Business Context / Problem
  • ข้อมูลคลินิก IPD อยู่ในฐานข้อมูล HAOS ทั้งหมด (TB_IPD_*/TBM_IPD_* ~55 ตาราง ที่สร้างตอนพัฒนา Smart IPD)
  • Smart IPD เข้าถึง HAOS 2 ทาง: HAOS API (Kiota) + ต่อ HAOS DB ตรง → ผูกกับ HAOS แยกไม่ได้
Business Value / Success Metrics
  • เปิดตลาดไป รพ. ที่ใช้ HIS เจ้าอื่น · ลดการพึ่งพา vendor เดียว
  • ตัวชี้วัด: เชื่อม HIS เจ้าใหม่ได้ภายใน X สัปดาห์/เจ้า โดยไม่ต้องแก้ core
Deliverables
  • 📋 BRD/SRS (ความต้องการ + stakeholder + business rules + gap/impact)
  • 🔧 System & Data Design (schema Smart IPD + ER diagram, HIS read/write contract, identity model)
  • 🔧 Migration & Cutover plan · 📋🔧 Conformance & UAT plan
Definition of Done

เอกสารผ่าน review จาก Business Owner, Tech Lead, DBA, ตัวแทนทีม HIS · รายการตาราง Smart IPD-owned + ข้อมูล HIS ได้รับการยืนยัน · พร้อมส่งต่อทีม Dev

Background

การ์ดตั้งต้น — lock เป้าหมาย/ขอบเขตเชิงธุรกิจก่อนลงเทคนิค

Sub-tasks

  • จัดทำ Stakeholder map + RACI
  • รวบรวม business requirements + objectives + success metrics
  • นิยาม scope/out-of-scope + ข้อจำกัด (กฎหมาย/PDPA/เวชระเบียน/การเงิน)
  • นัด review + sign-off

  • รายการ business requirements + objective + success metrics ที่วัดได้

SABA-5ออกแบบ Schema + Migration: Master/Config (TBM_IPD_*) SP 8Highเฟส 2

ในฐานะ SA ฉันต้องการ ออกแบบ schema ของ master/config IPD ในฐานข้อมูล Smart IPD เพื่อ ตัด master ออกจาก HAOS เป็นกลุ่มแรก (coupling ต่ำสุด)

Background

TBM_IPD_* ไม่ผูกข้อมูลผู้ป่วย (มีแค่ FK ผู้สร้าง/แก้ไข) → quick win

Sub-tasks
  • ออกแบบ ER diagram + DDL ของทุกตาราง TBM_IPD_*
  • เปลี่ยน FK UserCreated/Modified เป็น reference value
  • ระบุ business owner ของ master แต่ละชุด
  • ร่างสคริปต์ migration
  • ยืนยันว่าตัดออกจาก HAOS ได้
Acceptance Criteria
  • ER diagram + DDL ครบ (ภาคผนวก A.1)
  • ไม่มี FK ข้ามฐาน — ใช้ reference value
  • สคริปต์ migration + verification (counts/checksums)
  • ระบุ business owner + ยืนยัน HAOS ตัดได้

SABA-6ออกแบบ Schema + Migration: Transactional (TB_IPD_*) SP 13Highเฟส 2

ในฐานะ SA ฉันต้องการ ออกแบบ schema ของเอกสารคลินิก IPD ในฐานข้อมูล Smart IPD เพื่อ ให้ Smart IPD เป็นเจ้าของเนื้อหาการดูแลผู้ป่วยในทั้งหมด

Background

TB_IPD_* = บันทึกแพทย์/พยาบาล, คำสั่ง, eMAR, แผนการพยาบาล, รับผู้ป่วยใหม่, สรุปจำหน่าย, med reconcile, I/O records — ผูกแกน HIS ผ่าน AdmitId/PatientId/UserId

Sub-tasks
  • ออกแบบ ER diagram + DDL ของทุกตาราง TB_IPD_* (รวม extension/child)
  • เปลี่ยน FK ข้ามฐาน → reference value (ตาม SABA-7)
  • ออกแบบจัดการตารางผูกแน่น (TB_IPD_DISPENSING_* ↔ drug/finance)
  • ยืนยันกับธุรกิจว่าเอกสารใดต้องสำเนา/ส่งกลับ HIS
  • ร่างสคริปต์ migration + กลยุทธ์ integrity
Acceptance Criteria
  • ER diagram + DDL ครบทุกตาราง (ภาคผนวก A.2)
  • ไม่มี DB-level FK ข้ามฐาน — ใช้ HisAdmissionRef/HisPatientRef/HisUserRef
  • แนวทางจัดการตารางผูกการเงิน/ยา + migration + integrity
  • ยืนยันรายการเอกสารที่ต้องส่งกลับ HIS (โยง SABA-9)
  • ผ่าน review (DBA + Tech Lead)
Deliverables: ER diagram, DDL, migration script (transactional)
Dependencies: SABA-3, SABA-5, SABA-7 · DoD: design + script ผ่าน review

📎 ภาคผนวก A — ตารางที่ "ต้องมีใน Smart IPD" และ "ไม่จำเป็นต้องมีใน HIS"

ถูกสร้างตอนพัฒนา Smart IPD (อยู่ใน HAOS DB ปัจจุบัน) — เป็น domain ของ IPD จึงย้ายมาเป็นของ Smart IPD และตัดออกจาก HIS ได้

A.1 Tier 1 — Master / Config (TBM_IPD_*, ~22 ตาราง) — coupling ต่ำ ย้ายก่อน

FAIE Master + link tables: TBM_IPD_FOCUSES, TBM_IPD_FOCUS_CATEGORIES, TBM_IPD_FOCUS_SUBTYPES, TBM_IPD_ASSESSMENTS, TBM_IPD_ASSESSMENT_ABNORMAL_FINDINGS, TBM_IPD_INTERVENTIONS, TBM_IPD_EXPECTED_OUTCOMES, TBM_IPD_NURSE_NOTE_TEMPLATES (+ _ASSESSMENTS, _INTERVENTIONS, _EXPECTED_OUTCOMES)

Intake/Output Master: TBM_IPD_INTAKE, TBM_IPD_OUTPUT

Audit logs: TBM_IPD_FOCUS_LOGS, TBM_IPD_FOCUS_CATEGORY_LOGS, TBM_IPD_FOCUS_SUBTYPE_LOGS, TBM_IPD_ASSESSMENT_LOGS, TBM_IPD_INTERVENTION_LOGS, TBM_IPD_EXPECTED_OUTCOME_LOGS, TBM_IPD_NURSE_NOTE_TEMPLATE_LOGS, TBM_IPD_INTAKE_LOGS, TBM_IPD_OUTPUT_LOGS

ความผูกพันกับ HIS: มีแค่ FK UserCreated/Modified → แก้เป็น reference value

A.2 Tier 2 — Transactional (TB_IPD_*, ~33 ตาราง) — ผูก AdmitId/PatientId/UserId

Admission / Diagnosis: TB_IPD_ADMISSION_RECORDS (+ extension ~11), TB_IPD_DIAGS, TB_IPD_DOC_DIAGS, TB_IPD_PRE_DIAGS, TB_IPD_PROCEDURES, TB_IPD_HISTORY_PHYSICALS (+ _PROBLEM), TB_IPD_DISCHARGE_SUMMARIES, TB_IPD_CARDS

Progress notes / Orders: TB_IPD_DOCTOR_NOTES (+ _CANVAS, _PROBLEMS, _ACTIVITY_LOG), TB_IPD_DOCTOR_PROBLEMS, TB_IPD_NURSE_NOTES, TB_IPD_NURSE_FOCUS_ITEM, TB_IPD_ORDER_NOTES, TB_IPD_ORDER_NOTE_DETAILS, TB_IPD_ORDER_HOLD_LOGS, TB_IPD_ORDER_ACTIVITY_LOGS

eMAR: TB_IPD_ORDER_PLANS, TB_IPD_ORDER_PLAN_SCHEDULES, TB_IPD_ORDER_PLAN_ACTIVITIES

Nursing Care Plan: TB_IPD_PATIENT_NURSING_CARE_PLANS (+ _EXPECTED_OUTCOMES), TB_IPD_PATIENT_NURSING_PROGRESS_NOTES (+ _ASSESSMENTS, _INTERVENTIONS, _EXPECTED_OUTCOMES)

Intake/Output · Pharmacy · Finance ref: TB_IPD_INTAKE_OUTPUT_RECORDS, TB_IPD_DISPENSING_REQUEST, TB_IPD_DISPENSING_VERIFICATION, TB_IPD_MED_RECONCILE (+ _ITEM, _SIGNATURE, _AUDIT_LOG), TB_REF_FINANCES_IPD_RIGHTS ⚠️ ผูกการเงิน

A.3 Smart IPD-owned อื่น

  • Users, UserAccounts, RefreshTokens (auth — อยู่ใน Smart IPD DB แล้ว)
  • TB_PATIENT_TAGS (ปัจจุบันใน HAOS DB → ควรย้ายเข้า Smart IPD)
  • AI Chat cache, DICOM file index
TB_VITAL_SIGNS เป็นแกน HIS (ไม่ใช่ TB_IPD_*) — โดยปริยายคงอยู่ที่ HIS เว้นแต่ตกลงให้ Smart IPD รับเป็นเจ้าของ (ดู SABA-9)
