Smart IPD — การวิเคราะห์ API และความเป็นเจ้าของข้อมูล

API & Data Ownership Analysis — วิเคราะห์ว่า Smart IPD เรียก API อะไรบ้าง (ของตัวเอง vs HAOS) และข้อมูลใดสามารถนำมาเป็นของตัวเองได้ เพื่อลดการพึ่งพา HAOS API

อ้างอิงจาก codebase จริง 2 ฝั่ง: ayodia-haos-smart-ipd\src\backend และ top-panacea (HAOS HIS)  •  ฐานข้อมูล API: SmartIPD API Grouping.csv — 362 endpoints / 33 controllers

1. สรุปผู้บริหาร (Executive Summary)

  1. Smart IPD เป็น "hybrid proxy" — ปัจจุบันเกือบทุก endpoint proxy ไปยัง HAOS โดยมี local DB ของตัวเองแค่ 3 ตาราง (เรื่อง auth เท่านั้น) คือ Users, UserAccounts, RefreshTokens
  2. ข้อมูลคลินิกของ IPD ทั้งหมดอยู่ในฐานข้อมูล HAOS ในรูปตาราง TB_IPD_* (transaction) และ TBM_IPD_* (master/config) — ตารางเหล่านี้ ถูกสร้างตอนพัฒนา Smart IPD ตามที่เข้าใจไว้ถูกต้อง (พบ ~55 ตาราง)
  3. Smart IPD เข้าถึง HAOS ผ่าน 2 ช่องทาง:
    • HAOS API — Kiota-generated client (HAOS.SmartIPD.Libs.HaosClient)
    • เชื่อม HAOS DB ตรง — connection string HaosConnection ผ่าน HaosDatabaseService / Dapper / Stored Procedure
  4. โอกาสลดการพึ่งพา HAOS API = การ "ยึดครอง" ตาราง TB_IPD_* / TBM_IPD_* มาเป็นของ Smart IPD เพราะเป็น domain ของ IPD โดยแท้ ไม่ใช่แกนกลางของ HIS
Tier 1 ยึดได้ทันที — master/config (~22 ตาราง) Tier 2 ยึดได้แต่ต้องจัดการ FK — transaction (~33 ตาราง) Tier 3 ยึดไม่ได้ — แกน HIS

5. การวิเคราะห์: อะไรที่ Smart IPD เป็นเจ้าของเองได้

หลักการ: ข้อมูลที่ "เกิดและถูกใช้เฉพาะใน IPD workflow" → ควรเป็นของ Smart IPD ส่วนข้อมูล "แกนกลางของ HIS / ใช้ร่วมหลายระบบ (OPD, ER, billing)" → ต้องอยู่กับ HAOS

Tier 1 ยึดได้ทันที — Master/Config (แทบไม่ผูกข้อมูลผู้ป่วย)

ตาราง TBM_IPD_* เป็น master data/template ที่ไม่มี FK ไปยังผู้ป่วย/admission (มีแค่ FK ผู้สร้าง/แก้ไข = TB_USERS ซึ่งแก้ได้ด้วยการเก็บ user id แบบ value)

กลยุทธ์การแยก (Tier 2)

  • ย้ายตาราง TB_IPD_* ไปไว้ในฐานข้อมูล Smart IPD (haos_smartipd)
  • เก็บการอ้างอิงแกน HIS เป็น ค่าอ้างอิง (AN / AdmitId / PatientId / UserId) ไม่ใช่ DB-level FK ข้ามฐาน
  • ดึงบริบทผู้ป่วย (ชื่อ/ward/แพทย์) จาก HAOS แบบอ่านอย่างเดียว ตอนแสดงผล
  • ⚠️ Dispensing ผูกกับ TB_FINANCES (billing) — เชื่อมระบบการเงิน ควรแยกเป็นเฟสท้าย ๆ

ผลที่ได้: endpoints กลุ่ม clinical_documentation + orders_and_care_plan ส่วนใหญ่ (>150 เส้น) เปลี่ยนเป็น local ได้  ·  ความเสี่ยง: ปานกลาง

6. สรุปแผนเชิงปฏิบัติ (Recommended Roadmap)

เฟสขอบเขตความเสี่ยงผลลัพธ์
เฟส 1ย้าย TBM_IPD_* (Tier 1) + TB_PATIENT_TAGS เข้า haos_smartipdต่ำเลิกพึ่ง HAOS API: FAIE master data + I/O master + tags
เฟส 2ย้ายเอกสารคลินิกที่ผูกแค่ AdmitId/UserId (DoctorNotes, NurseNotes, FocusItems, NursingRecord, AdmissionRecord, HistoryPhysical, DischargeSummary)ปานกลางกลุ่ม clinical_documentation เป็น local
เฟส 3ย้าย OrderNote / OrderPlan(eMAR) / IntakeOutput recordsปานกลาง–สูงกลุ่ม orders_and_care_plan เป็น local
เฟส 4Dispensing / MedReconcile (ผูก billing/drug master)สูงพิจารณารายกรณี — อาจคงเชื่อม HAOS บางส่วน
ไม่ทำTier 3 ทั้งหมด–คงเรียกผ่าน HAOS API

สิ่งที่ต้องทำควบคู่ทุกเฟส

  1. ออกแบบ "anti-corruption layer" สำหรับอ่านบริบทผู้ป่วยจาก HAOS (อ่านอย่างเดียว)
  2. แทน DB-level FK ข้ามฐานด้วย reference value + validation ที่ระดับ application
  3. วางแผน migration ข้อมูลเดิมใน TB_IPD_* ออกจาก HAOS DB
  4. ทบทวน HaosDatabaseService ว่า query ตรงเส้นไหนเป็น TB_IPD_* (ยึดได้) vs TB_* แกนกลาง (ยึดไม่ได้)
